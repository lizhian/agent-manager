# Agent Manager 开发规范

## 1. 产品目标与基本原则

Agent Manager 是用于管理 Skills、MCP、Agent 配置和全局 `AGENTS.md` 的桌面工具。所有实现应从用户任务出发，保持界面简洁、信息清晰、操作可靠，并为后续模块扩展保留稳定边界。

- 优先保证数据安全、行为可预测和错误可恢复，再考虑视觉效果。
- 前端负责呈现与交互编排，Rust 后端负责文件系统、进程执行、持久化和系统能力。
- 业务规则只保留一份权威实现；禁止在前后端分别复制同一套状态转换逻辑。
- 新功能遵循现有结构与设计语言，不为单一场景引入通用框架。
- 默认使用中文界面文案；代码标识、协议字段和技术文档路径使用英文。

## 2. 技术栈与目录职责

- `src/routes/`：SvelteKit 页面，只负责页面组合、路由级状态和数据加载。
- `src/lib/components/`：可复用 UI 组件；组件按业务域组织，避免全局组件目录无限增长。
- `src/lib/features/`：前端业务模块，每个模块可包含 `api.ts`、`types.ts`、`state.svelte.ts` 和局部组件。
- `src/lib/styles/`：设计令牌、全局样式和通用布局规则。
- `src-tauri/src/commands/`：Tauri 命令层，负责参数校验、调用领域服务和转换错误。
- `src-tauri/src/domain/`：与界面无关的领域模型和业务规则。
- `src-tauri/src/infrastructure/`：文件系统、进程、序列化等外部能力。
- `src-tauri/src/lib.rs`：应用装配与命令注册，不放业务逻辑。

当前模块较小时允许合并文件；当单文件超过约 400 行、存在独立测试边界或被多个页面复用时，再按上述职责拆分。禁止仅为满足目录形式创建空抽象。

## 3. 架构规范

### 3.1 分层与依赖方向

依赖方向固定为：`页面/组件 -> 前端 API -> Tauri command -> 领域服务 -> 基础设施`。领域层不得依赖 Tauri、Svelte 或具体 UI 文案。

- Tauri command 使用可序列化 DTO 作为边界，不直接暴露内部路径对象或进程对象。
- 所有持久化修改应采用“准备新状态 -> 写入临时位置/备份 -> 原子替换 -> 同步派生状态 -> 失败回滚”的顺序。
- SQLite 数据库是应用设置和业务元数据的权威来源；软链、Skill 镜像索引、缓存和列表视图均视为可重建的派生状态。
- 跨平台系统行为集中封装。涉及 Windows、macOS、Linux 差异时必须提供条件编译或明确的不支持错误。
- 长耗时命令不得阻塞前端交互；界面必须显示进行中状态并防止重复提交。

### 3.2 状态管理

- 远端/后端状态以 Tauri command 返回结果为准，不用前端猜测写入结果。
- 临时输入、筛选、弹窗开关属于组件状态；跨页面且需要持久化的状态才进入共享 store。
- 每次写操作成功后，以后端重新读取的快照刷新界面，避免局部乐观更新造成漂移。
- 同一种状态只使用一个字段表达；禁止用多个布尔值组合出互斥视图，优先使用联合类型或枚举。

### 3.3 错误与日志

- 后端错误应包含动作、对象和根因，例如“更新来源 `owner/repo` 失败：npx skills 返回 …”。
- 用户可修复的错误直接在相关控件附近展示；破坏性操作失败保留弹窗和当前上下文。
- 不向界面泄露无意义的调试堆栈，也不吞掉底层错误。
- 禁止记录 token、凭据、完整环境变量或用户文档内容。

## 4. 代码规范

### 4.1 TypeScript 与 Svelte

- 开启并遵守严格类型检查；禁止无理由使用 `any`、非空断言和类型强转。
- 使用 Svelte 5 runes；派生状态使用 `$derived`，副作用使用 `$effect`，不要用副作用同步可计算状态。
- 组件 props、事件和 Tauri DTO 必须显式定义类型。
- 组件保持单一职责；业务请求集中在 `api.ts`，不要散落裸 `invoke`。
- 异步事件必须处理 loading、success、empty、error 四类状态。
- 文件名：组件使用 `PascalCase.svelte`，普通模块使用 `camelCase.ts`，类型名使用 `PascalCase`。

### 4.2 Rust

- 遵循 `rustfmt` 和 Clippy；公共 DTO 使用 `PascalCase` 类型与 `snake_case` Rust 字段，通过 serde 输出 `camelCase`。
- 可恢复错误返回 `Result<T, String>`（命令边界）或领域错误类型（内部），禁止在业务路径使用 `unwrap`、`expect`、`panic!`。
- 文件操作使用 `Path`/`PathBuf`，不手工拼接路径分隔符。
- 纯函数、校验函数和状态转换优先独立实现并单元测试。
- 外部命令必须捕获退出状态与 stderr，错误信息包含可行动上下文。

### 4.3 通用约束

- 默认使用 ASCII 代码标识；中文仅用于用户文案和中文文档。
- 注释解释“为什么”和边界条件，不复述代码。
- 不提交生成目录、用户配置、凭据、临时安装目录和构建产物。
- 单次改动保持聚焦，不顺手重构无关模块。

## 5. 前端设计规范

### 5.1 视觉语言

- 整体风格安静、专业、紧凑，适合频繁扫描和管理操作；避免营销页式大标题、装饰渐变和无信息价值的卡片堆叠。
- 使用中性灰作为主体，蓝色表达选中与主操作，绿色表达成功，琥珀色表达警告，红色只用于危险操作。
- 圆角不超过 `8px`；阴影克制，优先用边框和背景层级表达结构。
- 字号保持稳定，不随视口宽度缩放。应用提供特小、小、标准、大、特大五档，正文分别为 13、14、15、16、17px，标准档为默认值；标题、按钮和辅助文字通过同一组字号令牌同步调整。
- 间距基于 2px 网格，管理界面常用值为 2、4、6、8、12、16。禁止使用无信息价值的大面积留白。
- 所有颜色、字体、间距、圆角、阴影和动画时长应通过 CSS 变量定义。

### 5.2 信息密度

- 默认采用紧凑密度：标准按钮高度 `28px`，紧凑按钮和图标按钮 `24-26px`，输入框高度 `26-32px`。
- 按钮水平内边距通常为 `6-8px`，图标与文字间距为 `4px`；不得用大按钮填补布局空间。
- 顶部工具栏、筛选栏和面板之间的间距以 `6-8px` 为主；同一语义组内控件间距使用 `4px`。
- 单行列表高度通常控制在 `38-42px`，双行信息必须使用紧凑行高和辅助字号，不得因描述文本制造大行高。
- 面板标题栏高度通常为 `36-40px`，内边距以 `4-8px` 为主；表格、预览和弹窗内容区优先使用 `8-12px` 内边距。
- 提高信息密度时必须优先缩小外边距、内边距、控件间距和无效空白，禁止通过缩小正文、按钮、列表名称或描述字号来换取空间。
- 桌面管理界面的辅助文字不得小于 `10px`，按钮文字不得小于 `11px`，正文不得小于 `13px`；确需例外时必须经过实际截图验证其可读性。
- 页面首屏应优先展示可操作信息。顶部统计和操作栏不得形成独立的大块空白区域，也不得重复展示无决策价值的说明。
- 调整密度时必须同步检查点击目标、键盘焦点和文本可读性；紧凑不等于删除必要反馈或把文字压缩到不可辨识。

### 5.3 布局与响应式

- 桌面端使用稳定侧边导航和可滚动主工作区；内容区最大宽度根据任务密度决定，不强制卡片化。
- 重要操作放在页面标题行右侧；筛选和范围切换紧邻数据列表。
- 宽度不足时优先让工具栏换行、表格转列表、侧边栏收敛；禁止文字与按钮重叠。
- 所有固定格式控件定义稳定高度和最小宽度，加载文字、计数和图标不得导致布局跳动。

### 5.4 组件与图标

- 工具操作优先使用 Lucide 图标；不手绘已有图标。陌生图标提供 tooltip 和可访问名称。
- 二元设置使用开关或复选框；视图/范围切换使用分段控件；选项集合使用菜单；危险命令使用文本明确的按钮。
- 卡片只用于独立重复项、对话框和确有边界的工具，不在卡片内嵌套卡片。
- 空状态说明当前为什么为空，并提供唯一、明确的下一步操作。

## 6. 交互规范

- 每个操作必须有即时反馈：按钮按下态、进行中状态、成功结果或错误信息。
- 异步提交期间禁用冲突操作，但保留浏览和取消能力；按钮文案或 spinner 明确当前动作。
- 删除来源等破坏性操作必须二次确认，并展示影响范围与不可逆结果。
- 全局 Skill 开关点击后立即生效；仅锁定当前写操作，浏览、搜索和详情查看保持可用。
- 搜索应即时过滤名称、描述和来源；清空搜索后恢复原位置或合理焦点。
- 可排序列表使用专用拖拽手柄，拖动时明确显示上方或下方插入位置；长列表接近滚动边缘时应自动滚动。
- 键盘可访问：Tab 顺序符合视觉顺序，Enter/Space 符合控件语义，Escape 关闭最上层弹窗。
- 焦点样式清晰可见；弹窗打开时聚焦首个合理控件，关闭后焦点返回触发元素。
- 动画用于解释状态变化，时长通常 120-220ms；遵循 `prefers-reduced-motion`。
- 不使用仅靠颜色传达的信息；状态同时提供文字或图标。

## 7. 测试规范

### 7.1 测试分层

- Rust 单元测试：校验、catalog 规范化、选择状态转换、路径计算、描述解析和回滚相关纯逻辑。
- Rust 集成测试：使用临时目录验证 SQLite schema 与事务、稳定主键、安装镜像、全局软链同步、更新清理、旧 JSON/项目数据迁移与删除影响。
- 前端组件测试：加载、空状态、错误、筛选、选择、保存和确认弹窗。
- 端到端测试：至少覆盖安装来源、启用/禁用全局 Skill、更新单个来源、全部更新、查看详情和删除来源。

### 7.2 测试原则

- 测试行为与公开契约，不绑定无关 DOM 结构或私有实现。
- 每个缺陷修复必须先补充能复现问题的回归测试。
- 文件系统和数据库测试只写临时目录，不修改真实 `~/.agents`、项目目录或用户数据库。
- 外部 `npx skills` 通过接口/适配器替换为 fake；单元测试不得依赖网络。
- 时间、主目录和当前目录应可注入，保证测试可重复。

### 7.3 合并前检查

至少运行：

```bash
npm run check
npm run build
cargo fmt --check --manifest-path src-tauri/Cargo.toml
cargo test --manifest-path src-tauri/Cargo.toml
cargo clippy --manifest-path src-tauri/Cargo.toml -- -D warnings
```

涉及界面修改时，还需在桌面和最小窗口尺寸下人工或自动截图验证：无重叠、无截断、焦点可见、空/错/载入状态完整。

## 8. Skills Manager 领域约定

- Agent Manager 自有仓库位于 `~/.agent-manager/skills/`，权威数据库位于 `~/.agent-manager/agent-manager.db`。
- 数据库使用 `schema_migrations` 管理版本；通用设置写入 `app_settings`，Skills 来源和条目分别写入 `skill_sources` 与 `skills`。新增领域应建立明确业务表，不把结构化业务数据堆入设置表。
- 来源格式固定为 `owner/repo`，安全名为 `owner_repo`。
- 激活链接名固定为 `<source_safe>__<skill_name>`，避免不同来源的同名 Skill 冲突。
- 唯一激活目录为 `~/.agents/skills/`；不再提供项目级 Skill 激活能力。
- 更新来源采用镜像语义：保留仍存在 Skill 的启用状态，移除上游已删除 Skill，并清理其受管软链。
- 只管理 catalog 中可推导出的链接名；不得删除用户在目标目录内手工创建的其他文件。
- 安装、更新、删除和全局开关必须保持 catalog、镜像仓库和激活链接一致，失败时尽力回滚到旧状态。

## 9. AGENTS.md Manager 领域约定

- Markdown 片段存储在 SQLite 的 `agents_md_fragments` 表中；数据库是名称、正文、启用状态和排序的唯一权威来源。
- 启用片段按 `sort_order` 顺序组合，生成文件固定为 `~/.agents/AGENTS.md`；片段之间保留一个空行。
- 新增片段默认禁用。修改、删除、启用和排序成功后必须立即重新生成目标文件。
- 生成文件包含 Agent Manager 标记；不得覆盖没有该标记的用户文件而不保留备份。
- 首次接管已有 `~/.agents/AGENTS.md` 时备份为 `AGENTS.md.agent-manager-backup`；全部片段禁用后移除受管文件并恢复备份。
- 数据库写入或文件同步任一步骤失败时，必须恢复操作前的数据库和文件状态。
- 搜索只影响列表展示，不改变真实顺序；搜索状态下禁止拖拽排序，避免部分列表产生歧义。

## 10. 提交规范

- 使用 `feat:`、`fix:`、`docs:`、`test:`、`refactor:`、`chore:` 等 Conventional Commits 前缀。
- 一次提交只包含一个逻辑变更；提交前清理调试输出并运行相关检查。
- PR 描述包含用户可见变化、关键技术决策、测试结果和已知限制；界面变化附桌面与窄窗口截图。
